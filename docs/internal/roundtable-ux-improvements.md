## 全员共识

### 1. 供应商优先级控制
- **方案**: 拖拽排序 + 条件匹配（priority + condition 扁平结构）
- **技术**: `relay:priority:rules` 存储，内存 TTL 60s 缓存
- **KV 开销**: +1 GET（缓存命中时 +0），稳态 25-30 次不变
- **冲突处理**: 前端实时本地校验，冲突时卡片边框变红 + ⚠️ 徽标 + 文字提示弹窗

### 2. 供应商管理
- **方案**: 模板化创建 + 一键连通性测试
- **预置模板**: OpenAI、Anthropic、Google Gemini、阿里云通义、百度文心
- **交互**: 三步 Stepper 引导（选模板 → 配密钥 → 测试完成）
- **连通性测试**: Vercel Cron `/api/cron/probe`，每 30 分钟巡检，结果存 `relay:provider:health:{provider}`
- **KV 开销**: +0（后台任务）

### 3. 模型管理
- **方案**: 模型别名映射表 + 隐藏低频模型
- **技术**: `relay:models:aliases` 存储，内存缓存 5min，运行时零 KV 查询
- **交互**: 可增删行表格，支持 CSV 批量导入，宽松行高 + 斑马纹
- **KV 开销**: +0

### 4. 用量预警（轻量替代方案）
- **方案**: 每日 Cron 聚合报告（非实时预警）
- **技术**: `/api/cron/usage` 每日凌晨聚合 `relay:report:daily:{date}`
- **体验**: Admin 面板日期选择器 + 折线图趋势 + Webhook/邮件通知
- **KV 开销**: +0（Cron 写入，报告页读取不走请求链路）

## KV 预算分析

| 功能 | 每请求 KV 增加 | 备注 |
|------|---------------|------|
| 优先级规则 | +1 GET（缓存后 +0） | 内存 TTL 60s |
| 连通性测试 | +0 | Cron 写入 |
| 模型别名 | +0 | 内存缓存 |
| 用量报告 | +0 | Cron 写入 |

**峰值**: 26-31 次（首次请求或缓存过期）
**稳态**: 25-30 次（缓存命中）
**结论**: 在 +2 次约束内 ✅

## 设计方案

### 信息架构
固定侧边栏导航，四个模块：
1. **仪表盘**（默认页，关键状态概览）
2. **供应商管理**（供应商列表、优先级规则）
3. **模型配置**（别名管理、模型列表）
4. **监控与报告**（连通性状态、用量报告）

### 关键交互
- **优先级编辑器**: 卡片列表 + 拖拽，冲突时边框变红 + ⚠️ 徽标
- **模板创建**: 三步 Stepper 引导，线性无跳页
- **别名表格**: 宽松行高 + 斑马纹，操作列悬停显示
- **状态指示灯**: 颜色 + 文字标签（● 正常 / ⚠️ 超时 / ✕ 断开）
- **冲突提示**: 优先级覆盖用黄色警告条，配置冲突用红色阻断弹窗

### 移动端适配
- 供应商列表 → 卡片式布局
- 优先级排序 → 长按上下箭头
- 关键操作固定在卡片内

## 关键指标
- **首次操作无求助率**: 用户不看文档独立完成核心任务的比例
- **P95 配置生效延迟**: 用户改配置到生效的最慢时间（稳态 ≤60s）
- **KV 调用量**: 峰值 ≤31 次/请求

---

## 风险与缓解
1. **规则引擎复杂度**: 多规则交叉时 N² → 限制单租户最大 20 条规则
2. **Cron 巡检误报**: 瞬时抖动导致假 down → 连续 2 次失败才标记异常
3. **KV 单 key 限制**: `relay:config:all` 可能超 1MB → 拆分为多个 key
